- `app/api/...`
  - route handlers only
  - receive request
  - validate input
  - call service
  - return response

- `backend/services/...`
  - business logic
  - combine repositories + integrations
  - decide workflow

- `backend/repositories/...`
  - DB access only
  - select/insert/update/delete against Supabase tables

- `backend/integrations/...`
  - external APIs only
  - WhatsApp, Claude, later QuickBooks

- `types/...`
  - request/response/domain/external payload types

- `lib/...`
  - shared helpers
  - env, supabase client, phone/date utils

So when you build a feature, ask:
- is this DB access only? -> repository
- is this workflow/business logic? -> service
- is this HTTP endpoint? -> route
- is this third-party API? -> integration
- is this just utility? -> lib

# Practical naming rule

When you need a new feature, name files like this:

## DB-only
- `something-repository.ts`

Examples:
- `quotes-repository.ts`
- `pricing-items-repository.ts`

## Workflow/business
- verb-first or task-first service file

Examples:
- `create-quote-from-project.ts`
- `process-conversation-to-draft.ts`
- `send-whatsapp-message.ts`

## Route
- always under `app/api/.../route.ts`

Examples:
- `app/api/projects/[id]/route.ts`
- `app/api/review-drafts/process/route.ts`

For your current structure: **no**.

You already have the correct replacement:
- `lib/supabase/admin.ts`
- `lib/supabase/server.ts`

So do not make a random `db.ts` now.
That would blur responsibilities.

Use:
- `createAdminSupabaseClient()` inside repositories
- repositories inside services
- services inside routes

That is cleaner than one giant `db.ts`.
